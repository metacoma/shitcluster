# LLM_REPO_MAP.md

This file is a repository map for future local LLM/OpenHands sessions.

It describes the current repository structure and should be used as context before making changes.

## Repository purpose

The repository is a GitOps project (named "shitcluster") for managing a Kubernetes cluster. It uses KCL for defining infrastructure and workload configurations, Ansible for base node/cluster setup, and ArgoCD for continuous delivery. It also includes configurations for tmux sessions via KCL.

## High-level layout

| Path | Purpose | Notes |
| ---- | ------- | ----- |
| `ansible/` | Base infrastructure automation | Contains playbooks for Kubernetes (Kubespray), network, and storage setup. |
| `gitops/` | KCL-based GitOps configurations | Split into `infra/` and `workloads/`. |
| `tmux/` | Tmux session definitions | KCL files that generate tmux session configurations. |
| `secrets/` | Encrypted secrets | Uses SOPS for secret management. |
| `misc/` | Miscellaneous scripts | Utility scripts (e.g., Calico fix). |
| `vault-data/` | Vault configuration | Terraform files for HashiCorp Vault. |
| `Makefile` | Project-wide orchestration | Top-level commands for bootstrapping the cluster, ArgoCD, and Vault. |

## KCL layout

| Path | Type | Notes |
| ---- | ---- | ----- |
| `gitops/infra/` | Module | Infrastructure definitions; contains `kcl.mod`, `main.k`, `config.k`. |
| `gitops/workloads/` | Module | Application definitions; contains `kcl.mod`, `main.k`, `config.k`. |
| `gitops/workloads/apps/` | App Config | Individual `*.k` files for apps (e.g., `ssh_tunnel.k`, `istio.k`). |
| `tmux/` | Module | Tmux configuration; contains `kcl.mod`, `schema.k`, and `sessions/` directory. |
| `main.k` | Entrypoint | Root KCL file importing tmux sessions. |

## GitOps layout

- **Application directories**: `gitops/workloads/apps/` contains the source KCL for various applications.
- **Environment/Cluster structure**: Focuses on a single cluster (`mansion.shitcluster.io`).
- **Namespace structure**: Defined in `gitops/workloads/config.k`. Examples: `monitoring`, `observability`, `istio-system`, `knative-serving`.
- **Generated manifests**: KCL is used to stream YAML. manifests are likely piped to `kubectl apply` or consumed by ArgoCD.
- **Deployment files**: `Makefile` handles the high-level deployment flow (Kubespray $\rightarrow$ Vault $\rightarrow$ ArgoCD $\rightarrow$ Apps).

## Generated or read-only files

- `gitops/infra/tekton-pipelines.yml`: Contains markers indicating it might be partially generated or should be treated as a template.
- `kcl.mod.lock`: Lock files for KCL modules (automatically generated).

## Validation and build commands

Detected in `Makefile`:
- `make kubernetes`: Full cluster setup (via Ansible).
- `make vault`: Installs and initializes Vault.
- `make argocd`: Installs ArgoCD.
- `make longhorn`: Installs Longhorn storage.

Suggested, not confirmed KCL commands:
- `kcl fmt .`
- `kcl run .`
- `git diff --check`

## Useful search commands for future agents

- `find . \( -name 'kcl.mod' -o -name '*.k' \) | sort`: Find all KCL modules and files.
- `grep -r "schema " .`: Find KCL schema definitions.
- `grep -r "check:" .`: Find KCL validation checks.
- `grep -r "import " --include='*.k' .`: Find dependencies between KCL files.
- `grep -rE "DO NOT EDIT|generated|autogenerated|managed by" .`: Identify read-only files.

## Common change workflow

1. Read this file (`LLM_REPO_MAP.md`).
2. Inspect `git status --short`.
3. Search for existing similar KCL patterns in `gitops/workloads/apps/` or `tmux/`.
4. Edit the smallest possible KCL source files (e.g., `gitops/workloads/config.k` or specific app files).
5. Do not edit generated manifests directly.
6. Run detected validation commands (if environment is available).
7. Run `git diff --check`.
8. Show `git diff`.

## Notes for local LLM agents

- Prefer existing KCL schemas and patterns (especially in `tmux/schema.k`).
- Do not invent repository conventions.
- If validation needs secrets (SOPS), cluster access (kubeconfig), or private dependencies, report that clearly.
- Keep diffs minimal.
- Do not push unless explicitly asked.
